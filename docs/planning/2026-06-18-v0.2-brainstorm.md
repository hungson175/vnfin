## Q3 first — "test the published APIs for stability" (the heart of it)

There are **two distinct meanings**, both real, and we should do both:

### (3a) Stability of *our own* public Python API — no accidental breaking changes
A just-published library's #1 sin is silently renaming/removing a public symbol or changing a
result dataclass field/unit between releases. Proposed:
- **Public-surface snapshot test** — enumerate every public name (`vnfin.__all__`, each domain's
  `client/source/history/...`, every frozen-dataclass field name+type, every unit constant like
  `value_unit`) and assert it against a committed golden snapshot (`tests/api_surface.json`).
  Any add is fine; any **remove/rename/type-or-unit change fails CI** → forces a conscious SemVer
  decision instead of an accidental break. Cheap, fully offline, high leverage.
- **Deprecation policy** documented in `docs/stability.md` (SemVer: additive in minor, breaking only
  in major, one minor-release deprecation window).

### (3b) Stability of the *upstream* data-source endpoints we depend on
We are a thin client over ~15 third-party endpoints (SSI, VNDirect, VPS, Pinetree, CafeF, Fmarket,
BTMC, PNJ, Binance, Coinbase, World Bank, IMF, DBnomics, currency-api, Stooq). Any can change
shape, units, auth, or vanish at any time — and our users find out by getting broken data. We need
an **early-warning monitoring harness**, separate from CI unit tests:
- **Structured health-check runner** (`vnfin.health` or `scripts/healthcheck.py`) that probes each
  source and returns a typed `SourceHealth{source, reachable, schema_ok, value_sane, latency_ms,
  checked_at, note}`. `schema_ok` = expected JSON keys present; `value_sane` = e.g. FPT close in a
  plausible band, gold in plausible VND/lượng band. This is **monitoring**, not a pass/fail test.
- **Schema-drift assertions** — catch a provider silently changing keys/units *before* it corrupts
  user data (this is exactly how the VN-gold silver/unit bug would have been caught earlier).
- **Cross-source parity as a continuous check** — same FPT close across SSI/VNDirect/VPS within
  tolerance; divergence flags a broken/changed source.
- **Scheduled canary** — cron on the **PC** (has VN IPv4 reachability that this datacenter host
  lacks — IMF/some VN hosts 403 here) running the health runner daily/weekly, writing a timestamped
  JSON + a generated `STATUS.md` (per-source last-checked + health) so library users can see which
  sources are currently healthy. Alert via `tm-send`/file on breakage.
- Keep it **offline-unit-tested** (mock HTTP, synthetic fixtures) so the *harness* is CI-safe; the
  real network run is opt-in like `live_tests/`.

> Net: (3a) is a small high-leverage test + policy; (3b) is a proper monitoring harness — this is
> the single most valuable thing to build right after publishing, because it protects every user.

## Q2 — which data do we need more? (prioritized for long-term investors / macro analysts)

| Tier | Domain | Why it matters | Clean-room feasibility |
|------|--------|----------------|------------------------|
| **1** | **Corporate actions / dividends** (cash + stock dividends, splits, rights) | Total return is the whole game for long-term investors; we have adjusted prices but no explicit dividend events | Medium — needs source vetting (broker corporate-action endpoints) |
| **1** | **Company profile + shares outstanding** (sector/ICB, listing date, charter capital, shares O/S, market cap) | Needed to interpret fundamentals (per-share, P/E, market cap) consistently | Medium |
| **1** | **FX rates** (USD/VND, EUR/VND, CNY/VND, …) | Macro analysts need it; also converts world gold/crypto → VND; cleanest new domain | **High** — no-key sources exist (open ER APIs, SBV central rate) |
| **2** | **Ownership / foreign room** (major holders, room ngoại, state %) | VN-specific, important for liquidity/eligibility | Medium |
| **2** | **VN interest rates / bond yields** (SBV policy rate, interbank, govt yield curve) | Macro context; pairs with macro domain | Medium |
| **2** | **ETF support** (E1VFVN30, FUEVFVND… NAV/premium/holdings) | Popular passive vehicles; funds domain covers open-ended only | Medium |
| **3** | **Intraday/hourly prices** | Boss originally said "daily/hourly"; we shipped daily. Lower priority for long-term horizon | Medium |
| **3** | **News / earnings calendar** | Likely out of scope — unstructured + licensing risk | Skip for now |

**Builder's pick for v0.2 new-data:** start with **FX rates** (cleanest, no-key, immediately
useful, fast to TDD) and **corporate actions/dividends** (highest investor value). Company profile
+ shares-outstanding is a strong third if time allows. Ownership / rates / ETF → v0.3.

## Q1 — proposed v0.2 scope (bounded for an overnight autonomous run)

Sequenced, each step **TDD + reviewer sign-off + commit**, refactor only under green:

- **Phase A — Stability (do first, safest, all offline-testable):**
  1. Public-API surface snapshot test + `docs/stability.md` (SemVer/deprecation policy).
  2. Structured health-check harness (`SourceHealth` model + per-source probes + offline unit tests).
  3. Schema-drift + cross-source parity checks wired into the harness; generate `STATUS.md`.
  4. Scheduled canary design (cron-on-PC) — implement the runner + doc; actually scheduling the cron
     is a one-liner we can stage but only enable with a quick confirmation (writing crontab is the one
     externally-visible action — everything else is in-repo).
- **Phase B — New data:**
  5. **FX domain** — no-key failover (≥2 sources), unit = "VND per 1 unit of foreign currency",
     typed result + unit guard, synthetic fixtures, live cross-source parity test.
  6. **Corporate actions / dividends** — source-vet (clean-room, VNStock-excluded), design w/ reviewer,
     TDD adapter(s) + failover, document.
- **Phase C — close-out:** architecture review pass (reviewer), bump to **v0.2.0**, update README/api/
  roadmap, full green offline + live, tag (no secret-history risk this time — already scrubbed).

Independent domains (FX vs dividends vs health harness) → **git worktrees** to build in parallel,
merge only on green integration tests.
